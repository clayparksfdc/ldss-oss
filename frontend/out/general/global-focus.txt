19:T52a1,<p>Keyboard navigation of single page apps and large web interfaces can get complicated in a hurry. Use focus management appropriately to guide keyboard users through dynamic UIs.</p>
<p>Many users navigate web interfaces with only their keyboard, tabbing between focusable elements and using arrow keys to navigate interactive widgets such as menus. In dynamic user interfaces, the set of focusable elements on the screen changes as the user interacts with it.</p>
<p>When a user's action causes a UI change, such as an element appearing or disappearing from the screen, we can use focus management techniques to prevent the user from getting lost. When focus management is done right, it helps keyboard and screen reader users discover UI changes and easily reach the next logical step in their flow.</p>
<h2>Key Principles</h2>
<h3>Maintain Logical Focus Order</h3>
<p>The focus order should follow the visual and logical order of the content. Users should be able to tab through interactive elements, such as links, buttons, and form fields, in a predictable and intuitive sequence.</p>
<h4>Best Practices:</h4>
<ul>
<li>Use semantic HTML elements like <code>&#x3C;button></code>, <code>&#x3C;a></code>, and <code>&#x3C;input></code>, which have built-in focus behavior.</li>
<li>Avoid skipping focusable elements or creating confusing jumps in the focus sequence.</li>
<li>Use the <code>tabindex</code> attribute sparingly to adjust focus order, ensuring it aligns with user expectations.</li>
</ul>
<h3>Provide Visual Focus Indicators</h3>
<p>The Lightning Design System uses a global styling hook for all focus states, creating a visual cue for sighted keyboard users, users working in low visibility situations, and those with cognitive disabilities. Ensure any custom components utilize this global style for seamless integration.</p>
<h4>Best Practices:</h4>
<ul>
<li>Use the focus styling hook to define focus styles.</li>
<li>Avoid removing the focus outline unless you replace it with an equally visible alternative.</li>
</ul>
<h3>Manage Focus Programmatically</h3>
<p>Dynamic interfaces, such as modals, panels, popovers, and single-page experiences, often need programmatic control of focus. Managing focus helps users understand what changed on the screen and ensures that keyboard and assistive technology users aren’t left behind.</p>
<h4>When to Use Programmatic Focus</h4>
<p><strong>Opening Modals</strong></p>
<p>Move focus to the most meaningful element based on the modal’s purpose and structure. Refer to <a href="https://www.lightningdesignsystem.com/2e1ef8501/v/0/p/92a50f-global-focus/t/836dcb0951">Determining the Initial Focus When a Modal Opens</a> for specific guidance on multi-step modals, titled modals, and fallback behaviors.</p>
<h4>Closing Modals</h4>
<p>When the dialog closes, return focus to the element that triggered it. This helps users stay oriented in the flow they were working in.</p>
<h4>Error Handling</h4>
<p>If a form submission fails, move focus to the error message or the first invalid field. This immediately directs users to what needs attention.</p>
<h3>Dynamic Interface Updates</h3>
<p>For components that replace content on a page, such as guided flows, filtered results, or inline edit regions, move focus to the updated heading or the first meaningful control so that users understand what has changed.</p>
<h4>Examples</h4>
<p><strong>Focusing on an Element in a Modal</strong></p>
<pre><code>// Identify the appropriate target using your modal's initial-focus rules
const target = this.template.querySelector(
  '#modal-title, #step-subtitle, .first-interactive'
);

target?.focus();
</code></pre>
<p><strong>Returning Focus to the Trigger</strong></p>
<pre><code>// After the modal closes
triggerButton.focus();
</code></pre>
<p><strong>Focusing on an Error Message or Invalid Field</strong></p>
<pre><code>const error = this.template.querySelector('.slds-form-element__help');
error?.focus();
</code></pre>
<h3>Prevent Focus Traps</h3>
<p>Focus traps occur when users cannot navigate out of a particular section of the interface, such as a modal or dropdown menu.</p>
<h4>Best Practices:</h4>
<ul>
<li>Ensure that users can tab forward and backward through all focusable elements.</li>
<li>Provide an escape mechanism, such as the <code>Escape</code> key, to exit the trapped area.
<strong>Example:</strong> In a modal dialog, loop the focus back to the first element when users tab past the last focusable element.</li>
</ul>
<h3>Skip Navigation Links</h3>
<p>Provide a "Skip to Main Content" link at the top of the page to allow keyboard users to bypass repetitive navigation elements.</p>
<p><strong>Example:</strong></p>
<p><code>&#x3C;a href="#main-content" class="skip-link">Skip to Main Content&#x3C;/a></code></p>
<h3>Handle Focus Loss Gracefully</h3>
<p>Focus should never disappear or become "lost" when users interact with the application. This often happens in dynamic interfaces when elements are removed or replaced.</p>
<h4>Best Practices:</h4>
<ul>
<li>When removing an element that currently has focus, move focus to a relevant nearby element.</li>
<li>Ensure focus transitions are seamless and logical.</li>
</ul>

<h2>Lists and Tables</h2>
<p><strong>Component and Utility Class Examples</strong></p>
<h3>Removing an item</h3>
<p>When a row is removed from a list, via a button or menu on that list item, where to place focus depends on where the item sits within the list:</p>
<ul>
<li><strong>First row or any middle row:</strong> Place focus on the first focusable element in the next row.</li>
<li><strong>Last row in the list:</strong> Place focus on last focusable element of previous row.</li>
<li><strong>Only row in the list:</strong> Place focus on the list container or any content identifying the list as being empty.</li>
</ul>
<h3>Displaying more items in a list</h3>
<p>There are several ways to show more items in a list. The important aspect is to ensure that focus is never lost and that when new items are added, focus continues to move "downhill."</p>
<h4>"Show More" / "Show Less" toggle button</h4>
<p>Vertical Navigation is a good example of this strategy, as it uses markup for Expandable Sections inline with a list.</p>
<ul>
<li>User tabs through the items in the list.</li>
<li>If there is a "Show More" button, user tabs to this button after the last item in the list.</li>
<li>Pressing the "Show More" button reveals more items below the toggle, but focus remains on this button.</li>
<li>Pressing <strong>Tab</strong> on the "Show More" button moves focus to the first item after the button.</li>
</ul>
<h4>Disappearing "Show More" button</h4>
<p>Instead of a "Show More" / "Show Less" toggle, sometimes there may only be a single “Show More” button that goes away when pressed. If this is the case, place focus on the first new item that is added to the list.</p>
<h4>Lazy loading or infinite scrolling lists</h4>
<ul>
<li>If a list loads as a user scrolls down the page, it also needs to load as a user tabs through the items in the list. There should be no hiccups when tabbing through a lazy loading list. It is best to load new items as the user’s focus approaches the end of the list. Don’t wait until they reach the last item, or tab beyond the last item.</li>
<li>There must be some keyboard mechanism for moving beyond an infinitely loading list. This will prevent a keyboard-only user from getting stuck in a list that "never ends" when they want to move beyond the list. Options include but are not limited to:
A skip link before the list to take a user to the first focusable element after the list</li>
<li>A keyboard shortcut to move the user to the first focusable element after the list

<h2>Dialogs</h2>
<p>Dialogs appear in two primary forms—<strong>modal</strong> and <strong>non-modal</strong>—depending on how they interact with the rest of the application.</p>
<h3>Modal Dialogs</h3>
<p>A modal dialog appears above the main application, disabling interaction with the content behind it. Users must close the modal before returning to the underlying page.</p>
<p><strong>Examples:</strong> Modals, Welcome Mat</p>
<h3>Non-Modal Dialogs</h3>
<p>A non-modal dialog sits above the main application but doesn’t block it. Users can move focus back and forth between the dialog and the main interface.</p>
<p><strong>Examples:</strong> Docked Composers, Notification Cards, Panels, Popovers, Trial Bar</p>
<h3>Opening Dialogs</h3>
<p>Users open dialogs by activating a trigger, typically a button, by clicking or pressing the Enter or Space key. When the dialog opens, move focus into the dialog and trap it there so keyboard users stay within the dialog. Tabbing past the last focusable element should loop focus back to the first.</p>
<h4>Example: Dialog Trigger (LWC)</h4>
<pre><code>&#x3C;template>

      &#x3C;div class="slds-modal__container">

    &#x3C;div class="slds-backdrop slds-backdrop_open">&#x3C;/div>
  &#x3C;/template>
&#x3C;/template>
</code></pre>
<h3>Determining the Initial Focus When a Modal Opens</h3>
<p>To set initial focus where users expect it, use this decision flow. The goal is to give keyboard and assistive technology users a <strong>clear orientation</strong> the moment the modal becomes active. Follow these tiers to determine the most meaningful starting point.</p>
<h4>Tier 1: Multi-Step Modals</h4>
<p>To help users understand that a step has changed when a modal contains multiple steps and its subtitle updates, place focus on the step subtitle by making it temporarily focusable with tabindex="-1".</p>
<p><strong>Example: Step Subtitle (LWC)</strong></p>
<pre><code>&#x3C;h2 class="slds-text-heading_medium"
    tabindex="-1"
    aria-live="polite"
    id="step-subtitle">
  Step 2: Confirm Details
&#x3C;/h2>
</code></pre>
<pre><code>// After rendering a new step

</code></pre>
<h4>Tier 2: Standard Modal With a Title</h4>
<p>If the modal isn’t a wizard and contains a visible title, focus the modal title. This focus provides immediate context without skipping ahead to interactive controls.</p>
<p><strong>Example: Modal Title (LWC)</strong></p>
<pre><code>&#x3C;h1
  class="slds-modal__title slds-hyphenate"

  Edit Contact

Keyboard navigation of single page apps and large web interfaces can get complicated in a hurry. Use focus management appropriately to guide keyboard users through dynamic UIs.

Many users navigate web interfaces with only their keyboard, tabbing between focusable elements and using arrow keys to navigate interactive widgets such as menus. In dynamic user interfaces, the set of focusable elements on the screen changes as the user interacts with it.

When a user's action causes a UI change, such as an element appearing or disappearing from the screen, we can use focus management techniques to prevent the user from getting lost. When focus management is done right, it helps keyboard and screen reader users discover UI changes and easily reach the next logical step in their flow.

## Key Principles

### Maintain Logical Focus Order

The focus order should follow the visual and logical order of the content. Users should be able to tab through interactive elements, such as links, buttons, and form fields, in a predictable and intuitive sequence.

#### Best Practices:

- Use semantic HTML elements like `<button>`, `<a>`, and `<input>`, which have built-in focus behavior.
- Avoid skipping focusable elements or creating confusing jumps in the focus sequence.
- Use the `tabindex` attribute sparingly to adjust focus order, ensuring it aligns with user expectations.

### Provide Visual Focus Indicators

The Lightning Design System uses a global styling hook for all focus states, creating a visual cue for sighted keyboard users, users working in low visibility situations, and those with cognitive disabilities. Ensure any custom components utilize this global style for seamless integration.

#### Best Practices:

- Use the focus styling hook to define focus styles.
- Avoid removing the focus outline unless you replace it with an equally visible alternative.

### Manage Focus Programmatically

Dynamic interfaces, such as modals, panels, popovers, and single-page experiences, often need programmatic control of focus. Managing focus helps users understand what changed on the screen and ensures that keyboard and assistive technology users aren’t left behind.

#### When to Use Programmatic Focus

**Opening Modals**

Move focus to the most meaningful element based on the modal’s purpose and structure. Refer to [Determining the Initial Focus When a Modal Opens](https://www.lightningdesignsystem.com/2e1ef8501/v/0/p/92a50f-global-focus/t/836dcb0951) for specific guidance on multi-step modals, titled modals, and fallback behaviors.

#### Closing Modals

When the dialog closes, return focus to the element that triggered it. This helps users stay oriented in the flow they were working in.

#### Error Handling

If a form submission fails, move focus to the error message or the first invalid field. This immediately directs users to what needs attention.

### Dynamic Interface Updates

For components that replace content on a page, such as guided flows, filtered results, or inline edit regions, move focus to the updated heading or the first meaningful control so that users understand what has changed.

#### Examples

**Focusing on an Element in a Modal**

```
// Identify the appropriate target using your modal's initial-focus rules
const target = this.template.querySelector(
  '#modal-title, #step-subtitle, .first-interactive'
);

target?.focus();
```

**Returning Focus to the Trigger**

```
// After the modal closes
triggerButton.focus();
```

**Focusing on an Error Message or Invalid Field**

```
const error = this.template.querySelector('.slds-form-element__help');
error?.focus();
```

### Prevent Focus Traps

Focus traps occur when users cannot navigate out of a particular section of the interface, such as a modal or dropdown menu.

#### Best Practices:

- Ensure that users can tab forward and backward through all focusable elements.
- Provide an escape mechanism, such as the `Escape` key, to exit the trapped area.
**Example:** In a modal dialog, loop the focus back to the first element when users tab past the last focusable element.

### Skip Navigation Links

Provide a "Skip to Main Content" link at the top of the page to allow keyboard users to bypass repetitive navigation elements.

**Example:**

`<a href="#main-content" class="skip-link">Skip to Main Content</a>`

### Handle Focus Loss Gracefully

Focus should never disappear or become "lost" when users interact with the application. This often happens in dynamic interfaces when elements are removed or replaced.

#### Best Practices:

- When removing an element that currently has focus, move focus to a relevant nearby element.
- Ensure focus transitions are seamless and logical.

### Errors

- When the user attempts to save and is presented with an error:
If there are form element errors, place focus on the first input field with an associated error message. This makes it easy for the user to quickly correct the value.
- If an error popover appears, such as in the Docked Form Footer component, place focus inside the popover.
- If a page-level error appears, place focus on the error message.
- For error toasts, refer to the focus guidelines for Toasts.

## Lists and Tables

**Component and Utility Class Examples**

### Removing an item

When a row is removed from a list, via a button or menu on that list item, where to place focus depends on where the item sits within the list:

- **First row or any middle row:** Place focus on the first focusable element in the next row.
- **Last row in the list:** Place focus on last focusable element of previous row.
- **Only row in the list:** Place focus on the list container or any content identifying the list as being empty.

### Displaying more items in a list

There are several ways to show more items in a list. The important aspect is to ensure that focus is never lost and that when new items are added, focus continues to move "downhill."

#### "Show More" / "Show Less" toggle button

Vertical Navigation is a good example of this strategy, as it uses markup for Expandable Sections inline with a list.

- User tabs through the items in the list.
- If there is a "Show More" button, user tabs to this button after the last item in the list.
- Pressing the "Show More" button reveals more items below the toggle, but focus remains on this button.
- Pressing **Tab** on the "Show More" button moves focus to the first item after the button.

#### Disappearing "Show More" button

Instead of a "Show More" / "Show Less" toggle, sometimes there may only be a single “Show More” button that goes away when pressed. If this is the case, place focus on the first new item that is added to the list.

#### Lazy loading or infinite scrolling lists

- If a list loads as a user scrolls down the page, it also needs to load as a user tabs through the items in the list. There should be no hiccups when tabbing through a lazy loading list. It is best to load new items as the user’s focus approaches the end of the list. Don’t wait until they reach the last item, or tab beyond the last item.
- There must be some keyboard mechanism for moving beyond an infinitely loading list. This will prevent a keyboard-only user from getting stuck in a list that "never ends" when they want to move beyond the list. Options include but are not limited to:
A skip link before the list to take a user to the first focusable element after the list
- A keyboard shortcut to move the user to the first focusable element after the list

## Dialogs

Dialogs appear in two primary forms—**modal** and **non-modal**—depending on how they interact with the rest of the application.

### Modal Dialogs

A modal dialog appears above the main application, disabling interaction with the content behind it. Users must close the modal before returning to the underlying page.

**Examples:** Modals, Welcome Mat

### Non-Modal Dialogs

A non-modal dialog sits above the main application but doesn’t block it. Users can move focus back and forth between the dialog and the main interface.

**Examples:** Docked Composers, Notification Cards, Panels, Popovers, Trial Bar

### Opening Dialogs

Users open dialogs by activating a trigger, typically a button, by clicking or pressing the Enter or Space key. When the dialog opens, move focus into the dialog and trap it there so keyboard users stay within the dialog. Tabbing past the last focusable element should loop focus back to the first.
